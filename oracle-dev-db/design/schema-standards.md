# Schema Standards

## Overview

Oracle schema object standards for application development. Baseline target is **Oracle Database 19c**; syntax should remain forward-compatible with **26ai** where practical.

Read `project-prefix.md` before creating objects. Apply the project prefix to all objects listed there.

For PL/SQL package/procedure style, use `db/plsql/`. This file covers **tables, views, indexes, constraints, sequences, and column design**.

---

## Naming Conventions

### General Rules

- **Lowercase `snake_case`** for tables, columns, and non-constraint object names in DDL you author.
- **Unquoted identifiers** â€” Oracle stores them as uppercase internally; do not use double-quoted mixed-case names unless required (for example some database link or legacy object names).
- **Plural nouns** for table names: `orders`, `order_items`, `customer_statuses`.
- **Singular nouns** for column names: `order_id`, `status_code`, `email`.

### Constraint Names

Use the `db/` skill pattern with prefixed table names:

| Type | Pattern | Example |
|------|---------|---------|
| Primary key | `pk_{table}` | `pk_acme_orders` |
| Foreign key | `fk_{child}_{parent}` | `fk_acme_order_items_acme_orders` |
| Unique | `uq_{table}_{column}` | `uq_acme_customers_email` |
| Check | `ck_{table}_{column}` | `ck_acme_orders_amount` |

### Views

- Prefix: **`vw_`**
- Example: `vw_acme_open_orders`

### Sequences

- Use only when identity columns are not the right fit (see Primary Keys).
- Pattern: **`seq_{table}`** â†’ `seq_acme_invoice_numbers`
- Prefer identity columns for ordinary surrogate keys.

### Indexes

- Prefix: **`idx_`**
- Include column or purpose in the name when helpful: `idx_acme_orders_customer_id`
- When **more than one index** exists on the same table or view, append a numeric suffix: `idx_acme_orders_1`, `idx_acme_orders_2`, `idx_acme_orders_3`
- Number indexes in a stable order (document in the table script or project config if ambiguous).

## Primary Keys

### Default: Natural Keys

Use the **business key** as the primary key when it is stable, unique, and well understood.

Examples:

```sql
-- Lookup / reference table: code is the PK
CREATE TABLE acme_order_statuses (
    status_code   VARCHAR2(30)  NOT NULL,
    status_name   VARCHAR2(100) NOT NULL,
    CONSTRAINT pk_acme_order_statuses PRIMARY KEY (status_code)
);

-- Composite natural key on a junction table
CREATE TABLE acme_order_tags (
    order_id   NUMBER        NOT NULL,
    tag_code   VARCHAR2(30)  NOT NULL,
    CONSTRAINT pk_acme_order_tags PRIMARY KEY (order_id, tag_code),
    CONSTRAINT fk_acme_order_tags_acme_orders
        FOREIGN KEY (order_id) REFERENCES acme_orders (order_id)
);
```

### When to Use a Surrogate Key

Add a surrogate primary key (`id` column) only when:

- The natural key is wide, composite, or likely to change
- Multiple source systems supply conflicting natural keys
- ORM or integration patterns require a single numeric key
- High-volume child tables benefit from a narrow FK

### Surrogate Key Implementation

When a surrogate is needed:

```sql
CREATE TABLE acme_orders (
    order_id      NUMBER GENERATED BY DEFAULT ON NULL AS IDENTITY,
    order_number  VARCHAR2(50)  NOT NULL,
    -- â€¦
    CONSTRAINT pk_acme_orders PRIMARY KEY (order_id),
    CONSTRAINT uq_acme_orders_order_number UNIQUE (order_number)
);
```

**Identity column choice: `GENERATED BY DEFAULT ON NULL`**

| Option | Behavior | Use when |
|--------|----------|----------|
| `GENERATED ALWAYS` | Oracle always assigns; explicit values rejected | Maximum guard against manual IDs |
| `GENERATED BY DEFAULT` | Omit column â†’ generated; explicit value allowed; **NULL stays NULL** | Overrides for migrations |
| **`GENERATED BY DEFAULT ON NULL`** | Omit column â†’ generated; **NULL â†’ generated**; explicit value allowed | **Default for this project** â€” supports migrations, MERGE seeds, and bulk loads where `NULL` means "assign next" |

Standalone sequences (`seq_*`) remain acceptable when you need shared numbering across tables, gap-sensitive business numbers, or patterns identity columns do not cover.

### Primary Key Data Type

- Use bare **`NUMBER`** for surrogate keys so values can scale to Oracle's maximum precision.
- Size natural keys explicitly (`VARCHAR2(30)`, etc.).

### Gaps in Identifiers

Gaps from sequence cache, rollback, or restart are **acceptable**. Do not use `NOCACHE` to chase gap-free IDs.

## Standard Audit Columns

Every application table must include:

| Column | Type | Notes |
|--------|------|-------|
| `created_by` | `VARCHAR2(128) NOT NULL` | Application user or service account |
| `created` | `TIMESTAMP NOT NULL` | Default `SYSTIMESTAMP` |
| `modified_by` | `VARCHAR2(128)` | Set on update |
| `modified` | `TIMESTAMP` | Set on update |

```sql
CREATE TABLE acme_orders (
    order_id      NUMBER        GENERATED BY DEFAULT ON NULL AS IDENTITY,
    order_number  VARCHAR2(50)  NOT NULL,
    created_by    VARCHAR2(128) NOT NULL,
    created       TIMESTAMP     DEFAULT SYSTIMESTAMP NOT NULL,
    modified_by   VARCHAR2(128),
    modified      TIMESTAMP,
    CONSTRAINT pk_acme_orders PRIMARY KEY (order_id)
);
```

Populate `modified_by` and `modified` in application update logic or a centralized audit trigger/package â€” not only at insert time.

## Status and Lifecycle (Soft Delete)

Do not use `is_deleted` / `deleted_at` flags by default.

Use a **`status`** column that references a **lookup table** via foreign key:

```sql
CREATE TABLE acme_record_statuses (
    status_code   VARCHAR2(30)  NOT NULL,
    status_name   VARCHAR2(100) NOT NULL,
    CONSTRAINT pk_acme_record_statuses PRIMARY KEY (status_code)
);

CREATE TABLE acme_customers (
    customer_id   NUMBER        NOT NULL,
    email         VARCHAR2(255) NOT NULL,
    status_code   VARCHAR2(30)  NOT NULL,
    created_by    VARCHAR2(128) NOT NULL,
    created       TIMESTAMP     DEFAULT SYSTIMESTAMP NOT NULL,
    modified_by   VARCHAR2(128),
    modified      TIMESTAMP,
    CONSTRAINT pk_acme_customers PRIMARY KEY (customer_id),
    CONSTRAINT fk_acme_customers_acme_record_statuses
        FOREIGN KEY (status_code) REFERENCES acme_record_statuses (status_code),
    CONSTRAINT uq_acme_customers_email UNIQUE (email)
);
```

Seed status values (`ACTIVE`, `INACTIVE`, `CANCELLED`, etc.) with idempotent `MERGE` scripts â€” see `devops/idempotent-ddl-scripts.md`.

## Data Types

| Use case | Type | Notes |
|----------|------|-------|
| Text | `VARCHAR2(n)` | **Always specify `n`** â€” never default to 4000 without reason |
| Fixed codes | `VARCHAR2(n)` or `CHAR(n)` | Prefer `VARCHAR2` unless fixed width is required |
| Integers / IDs | `NUMBER` | Bare `NUMBER` for surrogates; precision on natural numeric keys when needed |
| Money / quantities | `NUMBER(p,s)` | **Case by case** â€” document precision per column |
| Booleans (19c) | `NUMBER(1,0)` + `CHECK` | `CHECK (col IN (0, 1))` â€” revisit when on 26ai SQL `BOOLEAN` |
| Event timestamps | **`TIMESTAMP`** | Default for date/time columns |
| Date-only (rare) | `DATE` | Only when time-of-day is never relevant |

Avoid `LONG`, `LONG RAW`, and unbounded bare `NUMBER` on financial amounts.

## Foreign Keys and Indexes

### Referential Actions

- Default: **no action** (restrict) â€” do not add `ON DELETE CASCADE` unless the use case requires it.
- Document any cascade or set-null exception in the script or project config.

### Index on Foreign Key Columns

**Create an index on every foreign key column** (or leading column of a composite FK).

> **Important:** Oracle does **not** automatically create an index on the **child** side of a foreign key. Oracle creates indexes automatically for **primary key** and **unique** constraints only. Without an index on the child FK column, deletes/updates on the parent table can lock the entire child table, and joins may perform poorly.

```sql
CONSTRAINT fk_acme_order_items_acme_orders
    FOREIGN KEY (order_id) REFERENCES acme_orders (order_id);

CREATE INDEX idx_acme_order_items_order_id
    ON acme_order_items (order_id);
```

When multiple indexes exist on `acme_order_items`, use numeric suffixes: `idx_acme_order_items_1`, `idx_acme_order_items_2`.

## Lookup Tables

Prefer **lookup tables + FK** over `CHECK (status IN (...))` for codes that may grow or need labels:

- Status, type, category, reason codes
- Seed with `MERGE` in install scripts
- Natural key on the code column (`status_code`, `type_code`)

Use `CHECK` constraints for invariant business rules that are not lookup-driven (non-negative amounts, valid date ranges).

## Tablespaces and Storage Clauses

- **Omit `TABLESPACE` and storage clauses** from application DDL â€” let the schema default tablespace apply.
- Strip storage/tablespace from extracted DDL checked into git (same idea as `db/devops/version-control-sql.md`).

## Schema Users and Privileges

Follow `db/design/data-modeling.md` security section:

- **Schema owner** account: holds objects; locked or not used for application connections
- **Application user**: `CREATE SESSION` plus explicit DML grants on required tables/views
- **Migration/install account**: DDL privileges as needed for script deployment

## Views

```sql
CREATE OR REPLACE VIEW vw_acme_open_orders AS
SELECT o.order_id,
       o.order_number,
       o.created
FROM   acme_orders o
JOIN   acme_order_statuses s ON s.status_code = o.status_code
WHERE  s.status_code = 'OPEN';
```

Use `CREATE OR REPLACE` in install scripts.

## Quoted Identifiers

Avoid double-quoted identifiers. Exception: integrating with legacy or database-link objects that were created quoted and cannot be renamed.

## Version Notes (19c vs 26ai)

| Topic | 19c (current default) | 26ai (upcoming) |
|-------|----------------------|-----------------|
| Identity columns | `GENERATED BY DEFAULT ON NULL AS IDENTITY` | Same |
| Boolean columns | `NUMBER(1,0)` + CHECK | May switch to SQL `BOOLEAN` later |
| `FETCH FIRST` / ANSI joins | Supported | Supported |
| Timestamp default | `TIMESTAMP` + `SYSTIMESTAMP` | Same |

Write new scripts for 19c; avoid 23ai-only syntax unless guarded with version checks.

## Common Mistakes

| Mistake | Why it is wrong |
|---------|-----------------|
| Assuming Oracle indexes FK columns automatically | Only PK/unique get implicit indexes; index the child FK yourself |
| `GENERATED ALWAYS` when loading migrated data | Blocks explicit key inserts; use `BY DEFAULT ON NULL` |
| `CHECK` instead of lookup for status | Harder to extend; inconsistent labels |
| Missing audit columns | Breaks project standard and traceability |
| Uppercase table names in new DDL | Conflicts with project lowercase convention |
| Editing APEX export SQL | Out of scope â€” never modify `apex/` or `f###.sql` |
